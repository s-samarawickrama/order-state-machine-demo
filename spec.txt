MediPick Enterprise Workflow Engine
Complete State Machine & Business Rules Specification
This document specifies the complete configurable workflow engine underlying the MediPick pharmacy pickup platform: five independent, metadata-driven finite state machines coordinated by a central workflow engine, the business rules governing every transition, the medicine availability/reservation model, and the principles governing the frontend that renders it.
1. Workflow Architecture Overview
MediPick's order journey is decomposed into five independent workflows rather than one monolithic state machine. Each workflow owns a distinct concern and is only loosely coupled to the others through conditions -- e.g. the order workflow will not enter PREPARING unless the prescription workflow reports APPROVED, but the two machines otherwise progress independently.

3.6 Restrictions
Customers can never approve their own prescription -- pharmacist- or doctor-role only.
Pharmacist/doctor approval is mandatory; there is no auto-approval path once a case reaches PHARMACIST_REVIEW.
OTC-only orders never instantiate this workflow at all -- ORDER_LIFECYCLE reads NOT_REQUIRED and skips straight to PHARMACY_REVIEW.
Poor image quality routes to reupload, not rejection, unless the customer abandons the reupload loop.

5.5 The OTP ≠ Payment Rule
This is a load-bearing rule for the whole architecture:
OTP proves: “this is the correct customer.”
Payment proves: “transaction obligations are complete.”
Consequently a customer can be OTP-verified before paying (generating the OTP requires no payment condition), but the pharmacy cannot execute complete_handover -- and therefore cannot release the medicine -- until payment.status equals PAID. If handover is attempted while unpaid, the transition is blocked with reason PAYMENT_REQUIRED even though identity has already been confirmed.

None of these dependencies are implemented as if/else branches in application code. Each is expressed as a condition object referencing another workflow's state path (e.g. {"prescription.status": "APPROVED"}), evaluated generically by the same condition evaluator used for every other rule.
8. Medicine Availability & Reservation Model
8.1 Why Not Full Inventory Management
MediPick deliberately does not implement exact stock-count inventory. Pharmacies maintain a medicine catalogue with a manually-updated availability_status (AVAILABLE, OUT_OF_STOCK, TEMPORARILY_UNAVAILABLE, DISCONTINUED) rather than tracked quantities. Full inventory management would turn MediPick into a pharmacy ERP system, which is outside the project's scope.
8.2 The Race-Condition Problem
Catalogue availability alone is insufficient once more than one customer can order the same low-stock item concurrently -- e.g. a medicine with 1 physical box remaining and two simultaneous orders. Cart addition never reserves stock (the same principle marketplace platforms such as Uber Eats use: adding to cart means “I am interested,” not “this item is mine”). The reservation only happens once the pharmacy actively confirms availability for a specific order.

9. Metadata-Driven Frontend Principles
The frontend is an internal workflow inspector / simulator, not a customer-facing ordering screen. Its governing rule:
// WRONG -- business logic hardcoded in the UI
if (state === "READY_FOR_PICKUP") showOTP();
 
// CORRECT -- UI renders whatever the metadata describes
contextFields.map(field => <DynamicField config={field} />)
Every field the Context Simulator renders (OTP status, payment settlement, prescription clarity score, and any future field such as insurance approval) is generated from the transition's context_fields metadata, not from a name the frontend recognizes. Role-based access is implemented as an ordinary condition (a role key checked the same way as any other field), not a special-cased permission branch, so adding a new role or a new gated field requires a metadata change only.
Available actions are always computed server-side and returned to the client; the frontend never independently decides whether an action is legal, only whether to render it and how to explain a validation failure returned by the engine.

11. Sequence Diagrams
The state diagrams in the sections above show what each workflow can do in isolation. The sequence diagrams below show the same rules from the opposite angle -- who calls what, in what order, and which cross-workflow gate each call has to pass -- for the three scenarios that matter most in demonstration.
11.1 Overall Order Journey (all three actors)
The end-to-end path a mixed order takes across Customer, Pharmacy, and the Workflow Engine, including both cross-workflow gates: prescription approval before preparation, and payment before handover.
11.2 OTC Order -- Happy Path
No Prescription Validation workflow is instantiated at all for a cart-only order; the engine goes straight from availability confirmation to preparation.
11.3 Prescription Order
Clarity-score branching happens before a pharmacist is ever involved -- only clarity scores of 20 or above reach PHARMACIST_REVIEW at all. Once approved, this scenario rejoins the same payment/OTP sequence as the OTC path.
11.4 Customer Ghosts the Pickup
Demonstrates the recovery path rather than a failure: no data is lost and no order is silently cancelled. NOT_COLLECTED is a distinct state from CANCELLED specifically so a pharmacy can still reopen the pickup window without re-running the entire order.
11.5 Mixed Order -- Cart Item + Prescription Item in ONE Order
This is the scenario the original business rules were written around: a customer can add a non-prescription item (Betadine) to their cart and upload a prescription (Amoxicillin, Cetirizine) in the same checkout. MediPick creates exactly one order containing both item sources -- it never splits into two separate orders, and the two per-item source tags (CART vs PRESCRIPTION) live inside that single order's item list.
The par / and block above is deliberate: prescription validation and cart-availability checking are independent branches of the same order, not a sequential dependency on each other. Both must clear before the single PHARMACY_REVIEW -> PREPARING transition fires -- the order does not proceed on partial clearance, and it does not fork into two orders that could complete at different times.
12. Final Architecture & Transition Execution Flow
The sections above specify what each workflow does. This section specifies how a single request actually moves through the system, end to end, and what happens inside the engine for every transition attempt regardless of which workflow it belongs to.
12.1 System Architecture
Every layer below has exactly one responsibility. The Workflow Engine box is intentionally the only place business conditions are evaluated -- everything above it is plumbing (routing, correlation, orchestration) and everything below it is side effects (notifications, chat, payment calls, security logging), never business logic itself.
Layer
Responsibility
FastAPI Gateway
Receives the request, generates/propagates the correlation (request) ID
Order Service
Orchestrates the lifecycle -- loads the order, calls the engine, persists the result
Workflow Engine
Pure decision: given (current_state, event, role, metadata), is this transition legal? Never sends messages, updates UI, or calls payment services itself
State Commit
Applies current_state -> next_state only after the engine says the transition is legal
Audit Event
Immutable record of the attempt -- created for both successes and failures
Action Executor
Dispatches the rule's actions[] (notification, chat, payment, security event) as independent, individually-tracked side effects
Action Telemetry
Records SUCCESS/FAILED per action -- a failed notification does not roll back the state commit
12.2 Transition Execution Flow
This is the same logic every workflow's engine runs -- Order Lifecycle, Payment, Pickup Verification, Prescription Validation, and Issue Management all share this one execution path; none of them has its own bespoke version of it.
Two properties fall directly out of this flow and are worth stating explicitly for the report:
State-first consistency: an invalid transition produces no state change, no committed actions, and no SUCCESS audit entry -- only a FAILED audit entry with the specific unmet codes. There is no partial-transition state.
Uniform failure reporting: a rule mismatch (NO_SUCH_TRANSITION), a role mismatch (ROLE_REQUIRED), and an unmet business condition (e.g. OTP_REQUIRED, PAYMENT_REQUIRED) all return through the same {code, message} shape. The frontend never needs a special case for “wrong role” versus “wrong data” -- both are just unmet entries in the same list.
